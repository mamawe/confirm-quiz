# Skill: 确认测验

## 触发语义
当用户准备开始编写代码、请求代码审查、准备合并代码，或提及"实现计划"、"实现笔记"、"确认测验"时触发。

## 能力边界
- **能做**：生成实现计划、强制记录实现偏差、生成确认测验报告并充当合并门禁。
- **不能做**：不替代用户做业务决策，不允许在用户未通过测验前自行合并代码。

## 核心执行纪律
本 Skill 包含 `anti-rationalization.md`（反借口表），Agent 在执行任何阶段的动作前，必须对照该表检查自身是否在寻找借口跳过步骤。

## 工作流：实现前、中、后

### 1. 实现前 (Pre-implementation)
- 生成 HTML 格式的实现计划。
- 优先呈现最可能修改的决策点（数据模型、类型接口、UX流程）。
- 机械性重构说明放在最底部。

### 2. 实现中
- 强制在项目根目录维护 `implementation-notes.md` 文件。
- 模板参考 `templates/implementation-notes.md`。
- 遇到边界情况偏离计划时，必须选择保守方案，并在"Deviations"下记录原因，然后继续。

### 3. 实现后
- 生成 HTML 格式的测验报告（模板参考 `templates/quiz-report.html`）。
- 报告包含：上下文、直觉解释、具体做了什么、底部测验题。
- **硬性门禁**：必须明确提示用户"请完成测验，只有全部答对后我才会执行合并/提交操作"。用户未明确全部答对前，拒绝执行 `git commit/push/merge`。
